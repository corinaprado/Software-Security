# Software-Security
# Practices for Secure Software Report

## Client Summary and Software Requirements

Artemis Financial is a financial consulting company that develops personalized financial plans for its clients, including retirement, savings, investments, and insurance. Because the company provides services through a public web application, it needed to strengthen its security to protect sensitive customer information during transmission. Artemis Financial asked Global Rain to modernize its software by implementing secure communication mechanisms, including HTTPS encryption and a file verification process using a SHA-256 checksum. The goal was to ensure data confidentiality, integrity, and secure communication while reducing potential security vulnerabilities.

## Identifying Security Vulnerabilities

One of the strengths of my vulnerability assessment was identifying weaknesses related to insecure communication and data integrity. I configured the application to use HTTPS with SSL/TLS instead of unencrypted HTTP and implemented a SHA-256 checksum endpoint to verify that transferred data had not been modified. I also reviewed the application for dependency vulnerabilities using OWASP Dependency-Check and addressed outdated libraries where appropriate.

Secure coding is essential because software vulnerabilities can expose sensitive customer information, disrupt business operations, and damage an organization's reputation. Applying secure coding principles reduces the likelihood of cyberattacks, helps organizations comply with industry standards, and increases customer confidence. For a financial services company such as Artemis Financial, protecting confidential financial data is critical to maintaining trust and ensuring business continuity.

## Challenges During the Vulnerability Assessment

The most challenging aspect of the vulnerability assessment was interpreting the results generated by automated security tools and determining which findings represented actual risks versus lower-priority recommendations. However, these tools were also extremely helpful because they identified vulnerable dependencies that could easily be overlooked during manual code reviews. Learning how to analyze vulnerability reports improved my understanding of software security and risk management.

## Increasing Layers of Security

I increased the application's security by implementing multiple layers of protection rather than relying on a single security mechanism. First, I enabled HTTPS using SSL/TLS certificates to encrypt communications between clients and the server. Next, I implemented a SHA-256 checksum endpoint that verifies the integrity of transferred data. I also updated vulnerable dependencies identified during the security scan and followed secure coding practices throughout the refactoring process.

In future projects, I would continue using automated vulnerability scanning tools such as OWASP Dependency-Check, static code analysis tools such as SonarQube, secure code reviews, and penetration testing to identify vulnerabilities. I would also continue following guidance from NIST Secure Software Development Framework (SP 800-218) and OWASP secure coding recommendations when selecting mitigation techniques.

## Ensuring Functionality and Security

After refactoring the application, I verified that the software continued to function correctly by rebuilding the project, testing the application endpoints, and confirming that HTTPS communication worked properly. I also tested the checksum endpoint to verify that it generated the correct SHA-256 hash values for data verification. After making security improvements, I reran the vulnerability assessment using OWASP Dependency-Check to confirm that no new dependency vulnerabilities had been introduced and that previous issues had been resolved.

## Resources, Tools, and Secure Coding Practices

Several resources and tools were valuable throughout this project. OWASP Dependency-Check was used to identify vulnerable third-party libraries, while Java's MessageDigest class was used to implement SHA-256 hashing. Spring Boot's SSL configuration enabled secure HTTPS communication using TLS certificates. I also relied on NIST publications, FIPS standards, and OWASP secure coding guidance to ensure that my implementation followed recognized industry best practices. These tools and standards will continue to be valuable for future software development projects that require secure application design.
